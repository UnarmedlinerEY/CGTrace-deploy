03 — Decision Log

The actual prioritization calls

Not just what shipped — which trade-off was chosen, why, and the one call that needed revisiting. A log that only keeps the calls that aged well isn't a real log.

Shipped the cheap proof before the flagship.
Why
Story 2 reused data that already existed — lowest job size, real value, fastest way to validate the concept before investing further.
Trade-off
The more dramatic, multi-client demo (Story 4) shipped two sprints later than originally framed as the flagship.
Lens
WSJF: low job-size + real value = ship first to de-risk. Scrum: same call, framed as backlog reordering on new information.
Pulled "least essential" work forward, against the original plan.
Why
Story 6 was scheduled last, but Story 4 needed a credible "no broker record" case to be convincing — a half-finished multi-client story now was worse than a fully-convincing one bundled into the same sprint.
Trade-off
Story 5 — the originally-planned finale — dropped out of the front of the queue. It still shipped, but in the final sprint, alongside Story 7 rather than as the centerpiece.
Shipped a narrow slice to test an idea, then expanded once it worked.
Why
Validate whether goal-tagging resonated before investing in a full model that might be the wrong cut.
Outcome
It validated — the scope later expanded from a 2-item slice to the full set.
Why it stays
A decision log that only keeps the calls that turned out right on the first try isn't a real log.

04 — Opportunity Backlog

Considered, and deliberately deferred

A raised idea that did not earn a roadmap slot — kept visible with the reasoning, because saying "not yet, and here's why" is the prioritization call that matters most. The original ask was a built-in timesheet/timer for accountants; only half of it survived review.

Xero WIP Practice-mgmt adjacent Discovery needed Off-wedge
Auto-billing narrative on job completion PARKED
Idea
On sign-off of a reconciliation, auto-draft the WIP line back to the Xero job — a suggested description and unit estimate ("Cost-base reconciliation · 47 parcels · FY24–25"), ready for the accountant to confirm.
Why it's tempting
Reinforces the "we live inside your Xero workflow" story and clears a slice of end-of-job admin.
Why deferred
Off the core wedge (cost-base reconciliation), and it edges into territory XPM's native timer and WIP already own. Rebuilding practice-management billing inside a CGT tool means competing with the platform we integrate with.
Dropped outright
The "timesheet / timer" half of the original ask — XPM already captures time. Only the billing-narrative half is worth keeping alive.
Validation gate
Do accountants bill CGT recon as time or as a fixed fee? If fixed-fee, "billing units" is the wrong primitive entirely. Needs discovery before it earns a sprint.
Constraint if it ships
Suggest-and-confirm only, never auto-commit — a wrong unit count pushed silently into WIP is worse than no feature.
05 — Outside This Epic

Same PO muscle,
a different backlog

Not part of the Autopilot build above — real experience from my current role, included because it's the muscle behind this product's cornerstone integration bet: pushing a reconciled figure into the tax return.

ATO MST IITR DSP DSP API validation
Case study · ATO return schemas → CGTrace's push-to-return
The schema literacy behind CGTrace's deepest integration bet

In my current role I work closely with Business Analysis and Dev on the ATO's IITR and related return schemas for a tax compliance suite — the Message Structure Table and validation-rule set every digital service provider builds returns against. That work is where you learn what the "push a figure into the return" last mile actually costs: the ATO publishes one baseline schema per return type, but each vendor implements it inside its own data model and its own API validation, so a single value has to satisfy three rulebooks at once. And the hard part is never the mandatory fields — it's the conditional-mandatory ones, where the trigger condition lives in another field's value and two careful readers can disagree.

Why this matters to CGTrace

CGTrace's push into the Xero return is mocked in this demo — but it's designed by someone who has walked that last mile for real. That's why the push is a mapped field register (ATO label ↔ return field ↔ pushed figure) behind a review gate, not a fire-and-forget API call — and why the scope deliberately stops at the one figure the schema actually needs, written onto the matching return job, rather than pretending a whole return can be round-tripped.
